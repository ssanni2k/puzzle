# Схема БД (Drizzle)

```sql
users
  id UUID PK
  username VARCHAR UNIQUE NOT NULL
  email VARCHAR UNIQUE NOT NULL
  password_hash VARCHAR NOT NULL
  created_at TIMESTAMP DEFAULT now()

puzzles
  id UUID PK
  user_id UUID FK → users
  title VARCHAR NOT NULL
  grid_cols INT NOT NULL
  grid_rows INT NOT NULL
  image_key VARCHAR NOT NULL           -- ключ в MinIO
  contours_key VARCHAR                 -- ключ метаданных контуров в MinIO
  status VARCHAR DEFAULT 'pending'     -- pending → processing → ready / error
  is_public BOOLEAN DEFAULT false
  completions_count INT DEFAULT 0
  created_at TIMESTAMP DEFAULT now()

progress
  id UUID PK
  user_id UUID FK → users
  puzzle_id UUID FK → puzzles
  state_json JSONB NOT NULL            -- позиции кусочков, фиксация
  completed BOOLEAN DEFAULT false
  updated_at TIMESTAMP DEFAULT now()
  UNIQUE(user_id, puzzle_id)

reports
  id UUID PK
  reporter_id UUID FK → users
  puzzle_id UUID FK → puzzles
  reason TEXT
  created_at TIMESTAMP DEFAULT now()
```

# API-эндпоинты

## Auth
- `POST /auth/register` — регистрация (username, email, password)
- `POST /auth/login` — вход (email/username + password), set httpOnly cookies
- `POST /auth/logout` — очистка cookies
- `POST /auth/refresh` — обновление access token

## Puzzles
- `GET /puzzles` — список публичных пазлов (пагинация, поиск по названию, фильтр мои/все)
- `GET /puzzles/:id` — карточка пазла + метаданные контуров
- `POST /puzzles` — загрузка изображения + создание пазла (multipart)
- `PATCH /puzzles/:id` — изменение видимости (is_public)
- `DELETE /puzzles/:id` — удаление пазла + связанных файлов и прогрессов

## Progress
- `GET /progress/:puzzleId` — получение сохранения
- `PUT /progress/:puzzleId` — сохранение состояния сборки
- `DELETE /progress/:puzzleId` — сброс прогресса

## Reports
- `POST /reports` — жалоба на пазл

## Profile
- `GET /profile` — данные профиля + статистика
- `PATCH /profile/password` — смена пароля

## WebSocket
- `WS /ws/progress/:puzzleId` — real-time синхронизация прогресса для авторизованных

# Генератор jigsaw-контуров

## Алгоритм
1. Для сетки `cols × rows` каждое ребро каждого кусочка получает тип: **tab** (выпуклость), **blank** (вогнутость) или **flat** (граница пазла).
2. Смежные кусочки имеют комплементарные рёбра: tab ↔ blank.
3. Контуры отрисовываются Bezier-кривыми, экспортируются как SVG path data + метаданные координат.
4. Sharp нарезает изображение на кусочки по маскам, сгенерированным из контуров.

## Формат выхода (PieceData)

```ts
interface PieceData {
  id: string;
  row: number;
  col: number;
  path: string;        // SVG path data для контура
  targetX: number;     // целевая позиция X на игровом поле
  targetY: number;     // целевая позиция Y на игровом поле
  offsetX: number;     // смещение tab/blank от прямоугольника
  offsetY: number;
  top: 'tab' | 'blank' | 'flat';
  right: 'tab' | 'blank' | 'flat';
  bottom: 'tab' | 'blank' | 'flat';
  left: 'tab' | 'blank' | 'flat';
}
```

# Drag & Drop модуль

Собственная реализация на Pointer Events API (унифицирует mouse + touch):
- `pointerdown` → захват кусочка (поднимает z-index)
- `pointermove` → перемещение по плоскости
- `pointerup` → snap-проверка:
  - Порог по расстоянию: 20 px от центра целевой ячейки
  - Порог по углу: 5° от целевого поворота
  - При успехе — анимация к целевой позиции (CSS transition), фиксация
  - При неудаче — остаётся в точке отпуска
- Touch-поддержка из коробки через Pointer Events
- Вращение кусочков учитывается в snap-логике

# Функциональные требования к MVP веб-приложения «Пазлы»

## 1. Каталог и доступ к пазлам

### FR-101. Просмотр публичных пазлов
- Любой пользователь (включая гостя) видит ленту / сетку публично доступных пазлов.
- Каждый элемент содержит миниатюру изображения, название, сложность (размер сетки) и количество успешных сборок.

### FR-102. Поиск и фильтрация
- Пользователь может искать пазлы по названию.
- Авторизованный пользователь может переключать режим отображения: «Все публичные» / «Только мои».

### FR-103. Пагинация
- Список пазлов подгружается постранично или через бесконечную прокрутку.

### FR-104. Гостевой доступ к сборке
- Гость может открыть любой публичный пазл и начать сборку без регистрации.

---

## 2. Регистрация и аутентификация

### FR-201. Регистрация
- Пользователь вводит уникальное имя (username), email и пароль.
- Пароль должен соответствовать минимальным требованиям безопасности (длина ≥ 8 символов).
- При успешной регистрации создаётся учётная запись, и пользователь автоматически входит в систему.

### FR-202. Вход
- Пользователь вводит логин (email или username) и пароль.
- При успехе сервер возвращает access + refresh JWT-токены в httpOnly cookie (защита от XSS).

### FR-203. Выход
- Пользователь может выйти из учётной записи; cookies очищаются, состояние аутентификации сбрасывается.

### FR-204. Доступ к защищённым функциям
- Только авторизованные пользователи могут создавать пазлы, управлять ими, сохранять прогресс на сервере и пользоваться личным профилем.

---

## 3. Создание и управление своими пазлами (авторизованный)

### FR-301. Загрузка изображения
- Пользователь выбирает файл в формате JPG или PNG, размером не более 10 МБ.
- Указывает название пазла и желаемую сложность (размер сетки: 3×3, 4×4 … до 10×10).

### FR-302. Фоновая генерация пазла
- После загрузки изображения система запускает асинхронную обработку (генерацию контуров кусочков).
- Пользователь видит статус обработки: «Ожидание» → «Готовится» → «Готов» / «Ошибка».
- Когда пазл готов, он автоматически появляется в списке «Мои пазлы».

### FR-303. Приватность по умолчанию
- Созданный пазл изначально является приватным и не виден другим пользователям.

### FR-304. Публикация пазла
- Владелец может сделать пазл публичным, явно подтвердив согласие (например, через чекбокс и модальное окно «Это изображение станет доступно всем, продолжить?»).

### FR-305. Изменение видимости
- Владелец может в любой момент переключить статус видимости (публичный ↔ приватный).

### FR-306. Удаление пазла
- Владелец может удалить свой пазл; при этом удаляются все связанные файлы и сохранённые прогрессы.

---

## 4. Игровой процесс сборки

### FR-401. Инициализация игрового поля
- При открытии пазла отображается поле фиксированного логического размера (например, 800×600 px), масштабируемое под размер окна.
- Все кусочки изначально размещены в случайных позициях за пределами поля, с небольшим случайным поворотом (±15°).

### FR-402. Перетаскивание кусочков
- Пользователь может захватывать любой кусочек мышью или пальцем (touch) и перемещать его по экрану.

### FR-403. Фиксация кусочка
- При отпускании кусочек проверяется:
  - Расстояние от его центра до центра целевой ячейки меньше порогового (например, 20 px).
  - Разница поворота относительно целевого менее 5°.
- Если условия выполнены, кусочек плавно анимируется к точной целевой позиции и фиксируется (становится не перемещаемым).
- В противном случае кусочек остаётся в точке отпуска.

### FR-404. Индикация прогресса
- Отображается счётчик: «Зафиксировано X из Y» кусочков.

### FR-405. Завершение пазла
- Когда последний кусочек зафиксирован, показывается сообщение / анимация поздравления.
- Прогресс по данному пазлу помечается как завершённый.

---

## 5. Сохранение и восстановление прогресса

### FR-501. Локальное сохранение для гостя
- При каждом перемещении кусочка состояние (идентификаторы кусочков, их позиции, фиксация) автоматически сохраняется в localStorage браузера.
- При повторном открытии того же пазла гость видит предыдущее состояние.

### FR-502. Серверное сохранение для авторизованных
- Для авторизованного пользователя прогресс дублируется на сервере: дебаунсированные запросы (раз в 3–5 секунд) отправляют текущее состояние сборки.
- WebSocket-канал дублирует критичные обновления (фиксация кусочка, завершение) в real-time.
- При входе в пазл клиент сначала пытается загрузить сохранение с сервера; если его нет — начинает с чистого состояния либо восстанавливает из localStorage.

### FR-503. Сброс прогресса
- Пользователь может вручную сбросить прогресс и начать сборку заново (кнопка «Сбросить»).
- При завершении пазла сохранение автоматически удаляется или помечается как завершённое, чтобы не занимать место.

---

## 6. Подсказки

### FR-601. Выбор кусочка для подсказки
- Пользователь может кликнуть по любому незафиксированному кусочку на поле или выбрать его из боковой панели, чтобы активировать для него подсказку.

### FR-602. Подсказка «Показать место»
- При активации на игровом поле кратковременно (2–3 секунды) подсвечивается контур целевой ячейки, куда должен быть установлен выбранный кусочек.

### FR-603. Подсказка «Показать оригинал»
- При активации на заднем плане плавно проявляется исходное изображение (полупрозрачное) на 3–5 секунд, после чего исчезает.
- Эта подсказка доступна в любой момент, не требует выбора кусочка.

### FR-604. Доступность подсказок
- В MVP подсказки бесплатны и доступны всем пользователям без ограничений.

---

## 7. Профиль пользователя и статистика

### FR-701. Просмотр профиля
- Авторизованный пользователь может открыть свой профиль, где отображается:
  - Имя пользователя.
  - Количество созданных пазлов.
  - Количество полностью собранных пазлов (за всё время).

### FR-702. Смена пароля
- Пользователь может изменить пароль, указав старый и новый пароль.

### FR-703. Список своих пазлов
- В профиле доступен список всех созданных пользователем пазлов с их статусом (приватный/публичный).
- Из этого списка можно перейти к управлению (изменить видимость, удалить).

---

## 8. Жалобы на контент

### FR-801. Отправка жалобы
- Любой авторизованный пользователь может нажать кнопку «Пожаловаться» на странице публичного пазла.
- Жалоба сохраняется в системе для последующей ручной модерации администратором (без интерфейса администратора в MVP, но запись в БД обязательна).

# Фазы реализации

## Фаза 1: Инфраструктура (3–4 дня)
- Инициализация монорепо (Turborepo + pnpm workspaces)
- Docker Compose: PostgreSQL, Redis, MinIO, Nginx
- Скелет Fastify-сервера + health check
- Скелет Svelte 5 клиента + Vite + роутер
- `packages/shared`: Zod-схемы для auth/puzzle/progress
- CI: GitHub Actions (lint, typecheck, test)

## Фаза 2: Auth (2–3 дня)
- Регистрация / вход / выход (JWT в httpOnly cookie + refresh token)
- Zod-схемы в shared, TypeBox-схемы на сервере
- Middleware авторизации на Fastify
- Формы на клиенте, authStore (runes)

## Фаза 3: Каталог пазлов + Upload (3–4 дня)
- CRUD пазлов: эндпоинты + Drizzle-запросы
- Upload изображения → MinIO
- BullMQ-джоба: Sharp resize → статус pending → enqueue generation job
- Лента пазлов: пагинация, поиск, фильтр публичные/мои
- Страницы: каталог, детальный просмотр

## Фаза 4: Генератор контуров (4–5 дней) — ключевая сложность
- Модуль `packages/jigsaw-generator`
- Алгоритм генерации комплементарных рёбер (tab/blank)
- Отрисовка Bezier-контуров, экспорт SVG path data
- Нарезка изображения на кусочки (Sharp/Canvas по маске)
- Сохранение метаданных + масок в MinIO

## Фаза 5: Игровой процесс (5–7 дней) — самый объёмный этап
- Компонент PuzzleBoard (Svelte 5)
- Drag & Drop модуль на Pointer Events
- Snap-логика с порогами (20 px, 5°)
- Анимация фиксации (CSS transition)
- Счётчик прогресса
- Экран завершения

## Фаза 6: Сохранение прогресса (2–3 дня)
- localStorage для гостей (автосохранение при каждом snap)
- HTTP debounce для авторизованных + WebSocket (fastify-websocket)
- Сброс прогресса
- Загрузка прогресса при входе в пазл

## Фаза 7: Подсказки + Профиль + Жалобы (2–3 дня)
- Подсказка «Показать место» (подсветка целевой ячейки)
- Подсказка «Показать оригинал» (полупрозрачный overlay)
- Профиль: мои пазлы, статистика, смена пароля
- Жалобы: POST /reports

## Фаза 8: Полировка + Деплой (2–3 дня)
- Responsive дизайн, мобильная адаптация
- Nginx конфигурация для прода
- E2E тестирование (Playwright)
- Деплой настройка

**Итого: ~23–32 дня (один разработчик)**
